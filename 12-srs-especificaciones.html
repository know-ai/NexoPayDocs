SRS — Software Requirements Specification

Especificación integral para llevar NexoPay desde MVP a producción con estándares de clase mundial: alcance, arquitectura, requerimientos, seguridad, pruebas y operación de alta disponibilidad.

1. Contexto

Objetivo, Alcance y Stakeholders

Este SRS establece la base contractual y técnica común entre clientes, inversionistas y desarrolladores.

🎯

Objetivo

Construir una plataforma fintech interoperable para pagos P2P, cobros comerciales y operación bancaria digital centralizada en Venezuela.

📦

Alcance

Aplicación cliente móvil (Flutter), panel administrativo web (React), backend transaccional (Python + FastAPI) y plataforma de datos/observabilidad.

🤝

Stakeholders

Usuarios finales, comercios, bancos aliados, regulador, inversionistas, equipo de operaciones, seguridad, producto e ingeniería.

Definición de éxito: MVP productivo con seguridad bancaria, tiempos promedio de transacción <3s, disponibilidad 99.95%+ inicial y escalamiento progresivo hacia 99.99%.
2. Arquitectura

Arquitectura Objetivo de Producción

Arquitectura modular para escalar por dominio funcional y mantener tiempos de entrega altos con calidad operacional.

Canales y Frontends

  • 📱Frontend Cliente: Flutter (iOS/Android), biometría, QR, PoS virtual, notificaciones.
  • 🖥️Frontend Admin: React + TypeScript para operaciones, soporte, conciliación y analítica operativa.
  • 🔐Autenticación: JWT de corta duración + refresh tokens + device binding + MFA para operaciones críticas.

Backend y Plataforma

  • 🐍Backend Core: Python 3.12 + FastAPI (async), Pydantic v2, SQLAlchemy, Alembic.
  • 📨Mensajería: Kafka para eventos de negocio, auditoría y notificaciones asincrónicas.
  • 🗄️Datos: PostgreSQL (ACID), Redis (cache/rate limit), almacenamiento de eventos y observabilidad.
  • ☁️Infra: Docker + Kubernetes, IaC con Terraform, despliegues Blue/Green o Canary.

5. Datos

Esquema de Base de Datos (Referencia)

Modelo relacional para transacciones críticas y trazabilidad financiera.

Entidades Core

  • 👤users: identidad, estado KYC, nivel de riesgo.
  • 🏦bank_accounts: cuentas vinculadas y estado de verificación.
  • 💸transactions: monto, moneda, referencia, estado, trazas.
  • 🧾ledger_entries: doble partida e integridad contable.
  • 🔐auth_sessions: sesión, dispositivo, expiración, MFA.
  • 📜audit_logs: evento, actor, origen, hash de integridad.

Lineamientos de Diseño de Datos

  • ✓
    Integridad:

    FKs, constraints y transacciones ACID en rutas de pago.

  • ✓
    Idempotencia:

    Llaves idempotentes por operación financiera.

  • ✓
    Auditoría:

    Bitácora inmutable de eventos operativos y de seguridad.

  • ✓
    Particionado:

    Particiones por fecha y estrategia de archivado regulatorio.

  • ✓
    Privacidad:

    Enmascaramiento de datos sensibles en entornos no productivos.

6. QA

Estrategia de Pruebas de Clase Mundial

Calidad por diseño: cobertura funcional, técnica, seguridad y resiliencia operacional.

Pruebas de Backend

Unitarias, integración, contrato API (OpenAPI), pruebas de idempotencia y concurrencia en endpoints críticos.

Pytest + Schemathesis

Pruebas Frontend

Flutter widget/integration tests y React component/e2e tests para journeys de negocio end-to-end.

Flutter Test + Playwright

Pruebas No Funcionales

Performance, carga, estrés, caos, seguridad ofensiva (SAST/DAST) y pruebas de recuperación ante fallos.

k6 + OWASP ZAP
Gate de release obligatorio: No se libera a producción si falla cobertura mínima, seguridad crítica o presupuesto de error del SLO.

8. DevSecOps

Despliegue de Alta Disponibilidad, Escalable y Mantenible

Pipeline CI/CD

  • 🔄Build, tests, SAST, DAST y análisis de dependencias en cada PR.
  • 📦Artefactos versionados y firmados para trazabilidad completa.
  • 🚀Canary deployment con rollback automático por métricas SLO.
  • 📈Observabilidad centralizada: logs, métricas, trazas y alertas accionables.

Alta Disponibilidad (HA)

  • 🌎Multi-AZ en producción para todos los servicios críticos.
  • 🧯Plan DR con failover validado mediante simulacros periódicos.
  • 🔐Gestión de secretos centralizada (KMS/Vault) con rotación automática.
  • 🛠️Runbooks operativos y playbooks de incidentes auditables.
